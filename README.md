## Overview

This workflow automates the full iOS release pipeline:

```
Push to main
    │
    ▼
Select Xcode 26
    │
    ▼
Checkout Code → Read Flutter Version → Setup Flutter
    │
    ▼
flutter pub get → Patch Pub Cache → pod install
    │
    ▼
Extract Version → Import Certificate → Install Profile
    │
    ▼
Build .xcarchive → Export .ipa
    │
    ▼
Save IPA as GitHub Artifact → Upload to App Store Connect
```

**Runner:** `macos-15` (required — iOS builds must run on macOS)  
**Trigger:** Push to `main` branch, or manually via `workflow_dispatch`

## Architecture & Pipeline Flow

| Step | Action | Key Detail |
|------|--------|------------|
| 1 | Select Xcode 26 | Uses `maxim-lobanov/setup-xcode@v1` to ensure the correct Xcode version |
| 2 | Checkout | Fetches the full repository at the triggering commit |
| 3 | Read Flutter version | Reads from `.flutter-version` file — keeps versions in code, not in the workflow |
| 4 | Setup Flutter | Installs the exact Flutter version on the runner |
| 5 | `flutter pub get` | Resolves and downloads all Dart/Flutter dependencies |
| 6 | Patch pub cache | In-place source patches for two incompatible packages (see details below) |
| 7 | `pod install` | Resolves native iOS (CocoaPods) dependencies; `Podfile.lock` is deleted first to force a clean resolution |
| 8 | Extract version | Reads `version:` from `pubspec.yaml` and uses `GITHUB_RUN_NUMBER` as the build number |
| 9 | Import certificate | Creates a temporary keychain, imports the `.p12` distribution certificate |
| 10 | Install profile | Installs the `.mobileprovision` provisioning profile |
| 11 | Build IPA | Archives with `xcodebuild` using manual signing, then exports to `.ipa` |
| 12 | Save artifact | Uploads the `.ipa` to GitHub Actions artifacts (7-day retention) |
| 13 | Upload to ASC | Uploads via `xcrun altool` using App Store Connect API Key authentication |
| 14 | Cleanup | Deletes the temporary keychain regardless of success or failure |

## Repository Secrets & Variables Setup

Navigate to your repository → **Settings → Secrets and variables → Actions**

### Secrets (encrypted, never logged)

| Secret Name | Description | How to Obtain |
|-------------|-------------|---------------|
| `CERTIFICATE_BASE64` | Base64-encoded Apple Distribution `.p12` certificate | See [Apple Developer Setup](#apple-developer-setup) |
| `CERTIFICATE_PASSWORD` | Password used when exporting the `.p12` | Set when exporting from Keychain Access |
| `KEYCHAIN_PASSWORD` | Arbitrary password for the temporary CI keychain | Choose any strong password — used only during the build |
| `PROVISIONING_PROFILE_BASE64` | Base64-encoded `.mobileprovision` file | Download from Apple Developer Portal |
| `APP_STORE_API_KEY_ID` | App Store Connect API Key ID (10-char string) | App Store Connect → Users and Access → Keys |
| `APP_STORE_ISSUER_ID` | App Store Connect Issuer ID (UUID) | Same page as above |
| `APP_STORE_API_KEY_BASE64` | Base64-encoded `.p8` private key file | Downloaded once when creating the API key |

### Variables (plain text, visible in logs)

| Variable Name | Description | Example |
|---------------|-------------|---------|
| `APP_APPLE_ID` | Numeric Apple ID of your app in App Store Connect | `6478291234` |

### Encoding Files to Base64

On **macOS / Linux:**

```bash
# Encode a .p12 certificate
base64 -i Certificates.p12 | pbcopy        # macOS — copies to clipboard
base64 -w 0 Certificates.p12               # Linux — prints single line

# Encode a provisioning profile
base64 -i profile.mobileprovision | pbcopy

# Encode an API key
base64 -i AuthKey_XXXXXXXXXX.p8 | pbcopy
```

On **Windows (PowerShell):**

```powershell
[Convert]::ToBase64String([IO.File]::ReadAllBytes("Certificates.p12")) | Set-Clipboard
```

Paste the output directly into the GitHub secret value field. Do **not** add newlines.

## Apple Developer Setup

### Step 1 — Create a Distribution Certificate

1. Open **Xcode → Settings → Accounts → Manage Certificates**
2. Click **+** → **Apple Distribution**
3. Xcode will generate a CSR, submit it to Apple, and install the certificate
4. Open **Keychain Access**, find the certificate under **My Certificates**
5. Right-click → **Export** → choose `.p12` format → set a strong password
6. Base64-encode the file (see above) and store as `CERTIFICATE_BASE64`

### Step 2 — Create an App Store Distribution Provisioning Profile

1. Go to [developer.apple.com/account](https://developer.apple.com/account)
2. Navigate to **Certificates, Identifiers & Profiles → Profiles**
3. Click **+** → **App Store Connect** → select your App ID
4. Select the Distribution certificate created above
5. Name it (e.g. `Enforcer App Store Profile`) — this name must match the `PROVISIONING_PROFILE_SPECIFIER` in the workflow
6. Download the `.mobileprovision` file
7. Base64-encode and store as `PROVISIONING_PROFILE_BASE64`

### Step 3 — Create an App Store Connect API Key

1. Go to [App Store Connect → Users and Access → Integrations → App Store Connect API](https://appstoreconnect.apple.com/access/api)
2. Click **Generate API Key**
3. Set **Access** to **Developer** (minimum required for uploads)
4. Note the **Key ID** → store as `APP_STORE_API_KEY_ID`
5. Note the **Issuer ID** (shown at the top of the page) → store as `APP_STORE_ISSUER_ID`
6. **Download the `.p8` file immediately** — it cannot be downloaded again
7. Base64-encode and store as `APP_STORE_API_KEY_BASE64`

## Local Setup & Running Locally

### macOS

macOS is the only platform that can natively produce an `.ipa`. Follow these steps to replicate the CI pipeline locally.

#### 1. Install Homebrew

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

#### 2. Install Flutter

```bash
# Read the required version from the repo file
FLUTTER_VERSION=$(cat .flutter-version)

# Option A: via FVM (Flutter Version Manager — recommended)
brew install fvm
fvm install $FLUTTER_VERSION
fvm use $FLUTTER_VERSION --force

# Option B: manual download
# Visit https://docs.flutter.dev/release/archive and download the version
# Unzip to ~/development/flutter and add to PATH
export PATH="$HOME/development/flutter/bin:$PATH"
```

#### 3. Install CocoaPods

```bash
sudo gem install cocoapods
# Or if using system Ruby issues:
brew install ruby
gem install cocoapods
```

#### 4. Install Xcode 26

Install from the [Mac App Store](https://apps.apple.com/us/app/xcode/id497799835) or via `xcodes`:

```bash
brew install xcodes
xcodes install 26.0  # installs the correct version
sudo xcode-select -s /Applications/Xcode_26.0.app
xcodebuild -version   # verify
```

#### 5. Install Dependencies

```bash
flutter pub get
cd ios && pod install --repo-update && cd ..
```

#### 6. Apply Pub Cache Patches

See the [Pub Cache Patching](#pub-cache-patching--explained) section below for full details. Quick local apply:

#### 7. Build the Archive Locally

```bash

xcodebuild \
  -workspace ios/Runner.xcworkspace \
  -scheme Runner \
  -configuration Release \
  -destination 'generic/platform=iOS' \
  -archivePath build/ios/archive/Runner.xcarchive \
  archive

xcodebuild \
  -exportArchive \
  -archivePath build/ios/archive/Runner.xcarchive \
  -exportOptionsPlist ios/ExportOptions.plist \
  -exportPath build/ios/ipa
```

#### 3. Apply Pub Cache Patches (Windows PowerShell)

Windows does not have `sed`. Use PowerShell's `(Get-Content)` / `Set-Content` approach:

```powershell
$pubCache = "$env:LOCALAPPDATA\Pub\Cache\hosted\pub.dev"

function Patch-File($path, $oldStr, $newStr) {
    if (Test-Path $path) {
        (Get-Content $path -Raw) -replace [regex]::Escape($oldStr), $newStr | Set-Content $path -NoNewline
        Write-Host "Patched: $path"
    } else {
        Write-Warning "File not found: $path"
    }
}

function Remove-Line($path, $pattern) {
    if (Test-Path $path) {
        $content = Get-Content $path | Where-Object { $_ -notmatch [regex]::Escape($pattern) }
        $content | Set-Content $path
        Write-Host "Removed lines matching '$pattern' in: $path"
    }
}

$dsBase = "$pubCache\dropdown_search-3.0.1\lib"

Patch-File "$dsBase\dropdown_search.dart" "textTheme.subtitle2" "textTheme.bodySmall"
Patch-File "$dsBase\dropdown_search.dart" "textTheme.subtitle1" "textTheme.titleMedium"
Patch-File "$dsBase\src\selection_widget.dart" "textTheme.subtitle1" "textTheme.titleMedium"
Patch-File "$dsBase\src\popupMenu.dart" "textTheme.subtitle1" "textTheme.titleMedium"
Patch-File "$dsBase\src\selection_widget.dart" "isAlwaysShown" "thumbVisibility"
Patch-File "$dsBase\src\properties\scrollbar_props.dart" "isAlwaysShown" "thumbVisibility"
Remove-Line "$dsBase\src\selection_widget.dart" "showTrackOnHover"
Remove-Line "$dsBase\src\properties\scrollbar_props.dart" "showTrackOnHover"
Remove-Line "$dsBase\src\selection_widget.dart" "hoverThickness"
Remove-Line "$dsBase\src\properties\scrollbar_props.dart" "hoverThickness"

$saPath = "$pubCache\simple_animations-4.2.0\lib\anicoto\animation_mixin.dart"

```

## Pub Cache Patching — Explained

### Why Patches Are Needed

The project depends on two packages that were published against older Flutter APIs that have since been changed or removed. Rather than modifying the source code of the app (which would require vendor-forking or version bumps), the build pipeline patches the cached package sources directly. This is a pragmatic approach for locked dependency graphs where upstream packages have not released fixes.

### dropdown\_search 3.0.1 Patches

This package was written for Flutter 2.x and uses several deprecated or removed `TextTheme` getters and `Scrollbar` parameters.

| Patch | Old API | New API | Reason |
|-------|---------|---------|--------|
| TextTheme rename | `textTheme.subtitle2` | `textTheme.bodySmall` | Renamed in Flutter 3.0 Material You migration |
| TextTheme rename | `textTheme.subtitle1` | `textTheme.titleMedium` | Same Flutter 3.0 rename |
| Scrollbar rename | `isAlwaysShown` | `thumbVisibility` | Parameter renamed in Flutter 2.9 |
| Scrollbar removal | `showTrackOnHover` | *(deleted)* | Removed entirely from `Scrollbar` widget |
| Scrollbar removal | `hoverThickness` | *(deleted)* | Removed entirely from `Scrollbar` widget |

### simple\_animations 4.2.0 Patches

Flutter changed the return type of `TickerMode.getNotifier()` from `ValueNotifier<bool>` to `ValueListenable<bool>` (the supertype). Since `ValueListenable` cannot be assigned to a `ValueNotifier` variable without a cast, the package fails to compile against modern Flutter.

| Patch | Change | Reason |
|-------|--------|--------|
| `animation_mixin.dart` | Adds `as ValueNotifier<bool>` cast | `TickerMode.getNotifier` return type narrowed in SDK |

### Applying Patches on Any OS

The patches are idempotent — running them multiple times on already-patched files is safe (the pattern simply won't match). You can verify a patch was applied by inspecting the file:

```bash
# Check if subtitle2 still exists (should print nothing after patching)
grep -n "subtitle2" ~/.pub-cache/hosted/pub.dev/dropdown_search-3.0.1/lib/dropdown_search.dart

# Check the cast was added
grep "as ValueNotifier" ~/.pub-cache/hosted/pub.dev/simple_animations-4.2.0/lib/anicoto/animation_mixin.dart
```

## ExportOptions.plist Configuration

The file `ios/ExportOptions.plist` controls how `xcodebuild -exportArchive` packages the `.ipa`. It must exist in the repository. A minimal App Store configuration:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>method</key>
    <string>app-store</string>

    <key>teamID</key>
    <string>45XP48S67P</string>

    <key>signingStyle</key>
    <string>manual</string>

    <key>provisioningProfiles</key>
    <dict>
        <!-- Replace with your actual bundle ID and profile name -->
        <key>com.yourcompany.yourapp</key>
        <string>Enforcer App Store Profile</string>
    </dict>

    <key>stripSwiftSymbols</key>
    <true/>

    <key>uploadSymbols</key>
    <true/>

    <key>compileBitcode</key>
    <false/>
</dict>
</plist>
```

Replace `com.yourcompany.yourapp` with your actual bundle identifier and `Enforcer App Store Profile` with the name of your provisioning profile.

## Workflow Step-by-Step Reference

### Step 1.5 — Select Xcode 26

```yaml
uses: maxim-lobanov/setup-xcode@v1
with:
  xcode-version: '26'
```

Pins the Xcode version so the build is deterministic regardless of which versions Apple pre-installs on the runner image. The `26` selector matches the latest `26.x.x` installed.

### Step 3 — Flutter Version from File

The `.flutter-version` file in the repository root contains a single line with the exact Flutter version, for example:

```
3.24.0
```

This means the SDK version is tracked in version control alongside the code, not embedded in the workflow YAML. To upgrade Flutter, update this file and commit.

### Step 6 — Version Extraction

```bash
VERSION=$(grep '^version:' pubspec.yaml | awk '{print $2}' | cut -d'+' -f1)
BUILD_NUMBER=$GITHUB_RUN_NUMBER
```

- `VERSION` is the semantic version from `pubspec.yaml` (e.g. `1.2.3`) — the part before `+`
- `BUILD_NUMBER` is the GitHub Actions run number — auto-incremented, globally unique per repository, and always increasing — which satisfies Apple's requirement that build numbers never repeat

### Step 7 — Certificate Import

A temporary keychain named `build.keychain` is created and configured as the default for the duration of the job. The certificate is imported with access granted to `codesign` and `xcodebuild`. This is isolated from the runner's system keychain to prevent any side effects.

### Step 9 — Build IPA

The `signing_override.xcconfig` file is generated at runtime with values derived from the installed provisioning profile and secrets. This avoids hardcoding signing configuration in the Xcode project, making the same `.xcodeproj` work for both local (Xcode-managed) and CI (manual) signing.

The build is two phases:

1. `xcodebuild archive` — compiles the app and creates a `.xcarchive` bundle
2. `xcodebuild -exportArchive` — packages the archive into a distributable `.ipa` using `ExportOptions.plist`

### Step 11 — App Store Connect Upload

Authentication uses the API Key method (recommended over username/password). The key file is written to `~/.appstoreconnect/private_keys/` where `altool` expects it:

```
~/.appstoreconnect/private_keys/AuthKey_<KEY_ID>.p8
```

`xcrun altool --upload-app` then authenticates using `--apiKey` and `--apiIssuer` flags.

## Versioning Strategy

Versions are sourced from `pubspec.yaml`:

```yaml
version: 1.2.3+45
```

| Component | Value | Source |
|-----------|-------|--------|
| Marketing version (CFBundleShortVersionString) | `1.2.3` | `pubspec.yaml` before `+` |
| Build number (CFBundleVersion) | GitHub run number | `$GITHUB_RUN_NUMBER` |

The `+45` suffix in `pubspec.yaml` is intentionally ignored by the workflow — `GITHUB_RUN_NUMBER` is used instead because it monotonically increases and is guaranteed unique, which App Store Connect requires.

To release a new version, update the `version:` field in `pubspec.yaml` and push.

## Artifacts & Retention

The `.ipa` is saved as a GitHub Actions artifact named:

```
app-release-<VERSION>-<BUILD_NUMBER>.ipa
```

- Retained for **7 days**
- Available under **Actions → your workflow run → Artifacts**
- The `if: always()` condition ensures the artifact is uploaded even if later steps fail, which is useful for debugging signing issues

## Troubleshooting

### ❌ `security: SecKeychainItemImport: Unknown format in import.`

The `CERTIFICATE_BASE64` secret is malformed. Re-export the `.p12` from Keychain Access and re-encode it:

```bash
base64 -i YourCert.p12 -o cert_b64.txt
# Paste the contents of cert_b64.txt into the secret (single line, no newlines)
```

### ❌ `error: exportArchive: No signing certificate "Apple Distribution" found`

Either the certificate hasn't been imported correctly, or the `CODE_SIGN_IDENTITY` in the workflow doesn't exactly match the certificate's CN. Run locally to verify:

```bash
security find-identity -v -p codesigning
```

The string after the SHA hash is the exact value to use in `CODE_SIGN_IDENTITY`.

### ❌ `error: No profiles for 'com.example.app' were found`

The bundle ID in the provisioning profile doesn't match `PRODUCT_BUNDLE_IDENTIFIER` in `project.pbxproj`. Also verify the profile name in `ExportOptions.plist` matches the actual profile name exactly (case-sensitive).

### ❌ `pod install` fails with `Unable to find a specification for...`

The CocoaPods spec repo may be stale. The workflow runs `--repo-update` which should handle this, but locally you can also run:

```bash
pod repo update
```

### ❌ Pub cache patches fail with `No such file or directory`

The package may be cached under a slightly different path. Verify:

```bash
ls ~/.pub-cache/hosted/pub.dev/ | grep dropdown_search
ls ~/.pub-cache/hosted/pub.dev/ | grep simple_animations
```

If the versions differ (e.g. `dropdown_search-3.0.2`), the patches are no longer needed for that version — or the package version has changed and the patches must be reviewed.

### ❌ `textTheme.subtitle1` still causing compile errors after patching

Verify the patch was applied successfully:


### ❌ `altool: No suitable application records were found` (App Store upload)

The app record must already exist in App Store Connect before you can upload. Create it at [appstoreconnect.apple.com](https://appstoreconnect.apple.com) under My Apps → New App.

### ❌ Build number already used

App Store Connect rejects re-used build numbers. Since the workflow uses `GITHUB_RUN_NUMBER`, this should never happen in CI. If you're uploading manually and hit this, increment the build number.

## Security Best Practices

- **Never commit certificates, profiles, or `.p8` keys** to the repository. All sensitive material must live exclusively in GitHub Secrets.
- The temporary `build.keychain` is always deleted in the cleanup step (`if: always()`), even on failure.
- The certificate `.p12` file written to disk during the job is not stored as an artifact and is cleaned up with the keychain.
- The App Store Connect API key is scoped to **Developer** role — the minimum permission required for uploads.
- Rotate API keys periodically via App Store Connect → Users and Access → Keys.
- Consider enabling **required reviewers** on the `main` branch to prevent unauthorized pushes from triggering App Store uploads.

## FAQ

**Q: Can I run this on a self-hosted runner?**  
Yes. The runner must be macOS with Xcode installed. Set `runs-on: self-hosted` or use a label like `runs-on: [self-hosted, macos, xcode-26]`. All other steps remain the same.

**Q: Why is `Podfile.lock` deleted before `pod install`?**  
On fresh CI runners, the pod source cache doesn't exist yet. Deleting `Podfile.lock` forces CocoaPods to re-resolve dependencies from scratch against `--repo-update`, avoiding stale lock file conflicts.

**Q: Why use `GITHUB_RUN_NUMBER` instead of the build number in `pubspec.yaml`?**  
`pubspec.yaml`'s build number (`+N`) is typically only incremented manually and would repeat across branches. `GITHUB_RUN_NUMBER` is automatically incremented by GitHub, is globally unique per repository, and never decreases — which satisfies Apple's strict requirement that `CFBundleVersion` must always increase.

**Q: Can I change the 7-day artifact retention?**  
Yes, change `retention-days: 7` in the upload artifact step. Free GitHub plans support a maximum of 90 days.

**Q: How do I trigger the workflow manually without pushing?**  
Go to **Actions → iOS Build & Upload to App Store → Run workflow** in the GitHub UI. The `workflow_dispatch` trigger enables this.

**Q: What if I want to build for TestFlight only and not App Store?**  
Change the `method` in `ExportOptions.plist` to `app-store` (TestFlight and App Store share the same upload method). To avoid submitting for review, simply don't submit the build in App Store Connect after it appears in TestFlight.

**Q: Why does the workflow use `xcrun altool` instead of `xcrun notarytool` or Transporter?**  
`altool` is available on the macOS runner without any additional installation and supports API key authentication. Note that Apple has deprecated `altool` for notarization (replaced by `notarytool`), but it remains fully supported for App Store **uploads** via `--upload-app`.
